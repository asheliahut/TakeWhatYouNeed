Id Type

A scalar type to represent your ID

The 'ID' scalar type represents a unique identifier, this field is not meant to be human readable. This field will always be a STRING.Â 

This can be a normal id or a uuid

Example: "5" or "ce53861b-97bc-425b-9521-72cf77913244"

String Type

A scalar type to represent your Strings

The 'String' scalar type represents a UTF-8 set of characters. Used to represent human readable text.

Example: "I am a string value"

Boolean Type

A scalar type to represent your Booleans

The 'Boolean' scalar type represents a value of true or false.

Integer Type

A scalar type to represent your Integers

The 'Integer' scalar type represents a non fractional whole number.

Example: 5

Float Type

A scalar type to represent your floats

The 'Integer' scalar type represents a non fractional whole number.

Example: 5.0

Extra Types

  • List Type
  • NonNull Type
  • Enum Type
  • + More

List Type

A special type to represent lists of types.

The 'List' type represents a set of types that are set for the list. This means a collection of a type. This can even be a list of any Scalar types.

Example: [Character]

NonNull Type

A special type to apply to another type to mark not nullable

The 'NonNull' type represents is represented by a ! it can be applied to any other type including List Types. It makes a type mandatory.

Example: Character! or [Character]!

Enum Type

A special type that is converted between a name and description

The 'Enum' type represents a name, description, and value. When using Enum Types for other inputs it will automatically let users use human readable input and map to non human readable values.

Interface Type

Interfaces are top level representation that can be used to define adherence.

interface Character {
    id: ID!
    name: String
    friends: [Character]
    appearsIn: [Episode]
}

Object Type

Objects are just like a model of your data and should be represented as such.

type Human implements Character {
    id: ID!
    name: String
    friends: [Character]
    appearsIn: [Episode]
    homePlanet: String
}
type Droid implements Character {
    id: ID!
    name: String
    friends: [Character]
    appearsIn: [Episode]
    primaryFunction: String
}
type Ewok implements Character {
    id: ID!
    name: String
    friends: [Character]
    appearsIn: [Episode]
    cuddliness: Int
}

Query Type

Queries are just the definition of how to retrieve data and what objects to retrieve.

type Query {
    hero(episode: Episode): Character
    human(id: String!): Human
    droid(id: String!): Droid
    ewok(id: String, name: String): Ewok
}

Arguments

{
  ewok(name: "Matthew Trask") {
      id,
      name,
      cuddliness
  }
}
{
  "data": {
    "ewok": {
        "id": "1",
        "name": "Matthew Trask",
        "cuddliness": 10
    }
  }
}

Mutation Type

Mutations are like Create, Update and Delete Calls. They alter data in some way.

type Mutation{
    addFriendToEwok(friendType: String!, friend: ID!)
}

Schema Type

Schema just defines the root query. This can define whether your API will send or receive data.

schema {
  query: Query
  mutation: Mutation
}

PHP GraphQL Libraries

  • There has only been two options for so long and neither have been what graphql strives for.
  • The spec has continued and the older libraries have fallen behind in compliance.

Webonyx/graphql-php

  • This was the first GraphQL library for PHP to follow spec guidelines.
  • It continues to this day to be a strong contender.
  • Huge following
  • Strong Documentation
  • Close to Spec
  • Long term usage
  • Not as close to spec as possible
  • Performance is lacking in internals

Pros

Cons

youshido-php/GraphQL

  • This library was inspired to be made in order to make a more PHP friendly representation of the spec
  • Medium Following
  • Medium Documentation
  • More PHP friendly
  • Long term usage
  • Maintainers are super slow
  • Performance used to be an issue
  • Not close to spec

Pros

Cons

digiaonline/graphql-php

  • This library was built out of the frustrations of lack of spec respect and timely updates while maintaining the PHP style.
  • Fast under the hood
  • SDL based schema definition not experimental
  • More PHP friendly
  • Close to Spec
  • It's an alpha build
  • low documentation

Start Building in PHP

use Digia\GraphQL\Language\FileSourceBuilder;
use function Digia\GraphQL\buildSchema;
use function Digia\GraphQL\graphql;

$sourceBuilder = new FileSourceBuilder(__DIR__ . '/star-wars.graphqls');

$schema = buildSchema($sourceBuilder->build(), [
    'Query' => [
        'hero' => function ($rootValue, $arguments) {
            return getHero($arguments['episode'] ?? null);
        },
    ],
]);

$result = graphql($schema, '
query HeroNameQuery {
  hero {
    name
  }
}');
